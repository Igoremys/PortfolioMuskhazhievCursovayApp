# Тестирование проекта YPhoto

В этом разделе описано, как в проекте YPhoto использовались три инструмента и подхода контроля качества: **тест-планы**, **JaCoCo** и **Postman**. Тестирование разделено на автоматические проверки мобильного приложения и серверной части, контроль покрытия кода, а также ручную/интеграционную проверку REST API.

## 1. Цели тестирования

Основные цели тестирования проекта:

- убедиться, что пользовательские сценарии приложения работают корректно: регистрация, вход, работа с профилем, публикация фотографий, постановка лайков, поиск пользователей и фотографий, оффлайн-режим с кэшированием;
- проверить бизнес-логику без запуска полного приложения там, где это возможно;
- защитить критичные серверные сценарии авторизации, публикации фото и управления лайками от регрессий;
- получить измеримое представление о покрытии серверного кода тестами;
- вручную проверить реальные HTTP-запросы к API в условиях, близких к использованию клиентским приложением.

## 2. Тест-планы

Тест-планы использовались как основной документ для структурирования проверок. Для каждого функционального блока мы заранее определяли:

1. **Что проверяется** — модуль, экран, API-метод или бизнес-правило.
2. **Предусловия** — наличие пользователя, роль пользователя, токен авторизации, подготовленные данные в БД.
3. **Шаги проверки** — последовательность действий пользователя или HTTP-запросов.
4. **Ожидаемый результат** — корректный ответ, изменение состояния, ошибка в негативном сценарии.
5. **Тип проверки** — модульная, интеграционная, ручная через Postman.
6. **Статус** — пройдено, требует исправления, требует повторной проверки.

### 2.1. Общий тест-план проекта

| Раздел | Что проверяли | Тип проверки | Инструменты |
| --- | --- | --- | --- |
| Маппинг данных | Преобразование DTO ↔ Entity ↔ Domain, обработка null-значений | Модульные тесты | JUnit, Kotlin tests |
| Состояния UI | Переходы Loading/Success/Empty/Error, обработка событий | Модульные тесты | JUnit, Turbine, coroutines test |
| Аккаунты | Регистрация, вход, роль USER, JWT, обновление профиля | Модульные и API-тесты | JUnit, Mockito, Postman |
| Безопасность | JWT, фильтр авторизации, доступ только к своим фото | Модульные тесты, ручная проверка API | JUnit, Spring Security Test, Postman |
| Управление фото | Создание, получение списка, обновление, удаление фото | Интеграционные/контроллерные тесты и Postman | JUnit, Spring Test, Postman |
| Лайки | Постановка/снятие лайка, защита от двойного лайка, счётчик | Модульные/контроллерные тесты и Postman | JUnit, Postman |
| Поиск | Поиск пользователей по имени/email, поиск фото по названию | Модульные тесты и Postman | JUnit, Postman |
| Оффлайн-режим | Кэширование фото в Room, работа без сети, синхронизация | Модульные тесты Android, ручное тестирование | JUnit, Turbine, ручной тест |

### 2.2. Пример тест-плана для авторизации

| ID | Сценарий | Предусловие | Шаги | Ожидаемый результат |
| --- | --- | --- | --- | --- |
| AUTH-01 | Регистрация нового пользователя | Email отсутствует в БД | Отправить `POST /api/auth/register` с email, именем и паролем | Создан аккаунт с ролью `USER`, возвращен JWT |
| AUTH-02 | Вход с корректными данными | Пользователь существует | Отправить `POST /api/auth/login` | Возвращен JWT и данные аккаунта |
| AUTH-03 | Вход с неверным паролем | Пользователь существует | Отправить `POST /api/auth/login` с неправильным паролем | Возвращается ошибка авторизации |
| AUTH-04 | Доступ к чужому профилю | Есть два пользователя | Пользователь A запрашивает `GET /api/profiles/{id}` пользователя B | Возвращаются только публичные данные профиля |
| AUTH-05 | Доступ без токена | — | Отправить защищённый запрос без заголовка `Authorization` | Возвращается `401 Unauthorized` |

### 2.3. Пример тест-плана для управления фото

| ID | Сценарий | Предусловие | Шаги | Ожидаемый результат |
| --- | --- | --- | --- | --- |
| PHOTO-01 | Публикация нового фото | Пользователь авторизован | `POST /api/photos` с title, description, imageUrl | Фото создано, возвращён статус `201 Created` |
| PHOTO-02 | Получение списка фото | В БД есть фото | `GET /api/photos` | Возвращается список всех фото |
| PHOTO-03 | Получение фото по ID | Фото существует | `GET /api/photos/{id}` | Возвращается объект фото |
| PHOTO-04 | Удаление своего фото | Фото принадлежит пользователю | `DELETE /api/photos/{id}` | Фото удалено, статус `204 No Content` |
| PHOTO-05 | Удаление чужого фото | Фото принадлежит другому | `DELETE /api/photos/{id}` | Возвращается ошибка доступа или `403 Forbidden` |
| PHOTO-06 | Валидация обязательных полей | — | `POST /api/photos` с пустым title | Возвращается `400 Bad Request` |

### 2.4. Пример тест-плана для лайков

| ID | Сценарий | Предусловие | Шаги | Ожидаемый результат |
| --- | --- | --- | --- | --- |
| LIKE-01 | Постановка лайка | Фото существует, пользователь авторизован | `POST /api/photos/{id}/like` | Счётчик лайков увеличен на 1, статус лайка = `true` |
| LIKE-02 | Снятие лайка | Лайк уже стоит | `POST /api/photos/{id}/like` (повторно) | Счётчик уменьшен на 1, статус лайка = `false` |
| LIKE-03 | Двойной лайк | Лайк уже стоит | Повторный запрос без снятия | Возвращается ошибка или игнорируется (UNIQUE constraint) |
| LIKE-04 | Лайк без авторизации | — | Запрос без токена | Возвращается `401 Unauthorized` |

### 2.5. Как тест-планы связаны с автотестами

Тест-планы не заменяли автоматические тесты, а задавали структуру проверок. После описания сценариев мы переносили стабильные и повторяемые проверки в код:

- в `app/src/test/java/com/example/portfolioapp` находятся тесты клиентской логики: мапперы DTO/Entity/Domain, состояния ViewModel, работа PortfolioRepository, оффлайн-кэширование;
- в `server/src/test/java/com/example/portfolioapp` находятся серверные тесты: авторизация, профили, CRUD фото, лайки, поиск, безопасность;
- сценарии, где важно проверить полный HTTP-запрос, заголовки и тело ответа, дополнительно выполнялись вручную через Postman.

## 3. Автоматическое тестирование

### 3.1. Тесты мобильного приложения

В Android-модуле используются unit-тесты на Kotlin/JUnit. Они проверяют бизнес-логику без запуска UI:

- корректность маппинга `PhotoDto → PhotoEntity → Photo`;
- обработку `null`-значений в мапперах;
- состояние `PhotoViewModel`: переходы `Loading → Success / Empty / Error`;
- реакцию на успешное создание/удаление фото;
- оптимистичное обновление лайка в `PortfolioRepository`;
- работу с Flow и StateFlow через Turbine;
- фильтрацию фото по `authorEmail` для экрана профиля;
- работу оффлайн-режима при отсутствии сети.

Запуск тестов приложения:

```bash
./gradlew :app:testDebugUnitTest
``

### 3.2. Тесты серверной части

Серверный модуль тестируется на Java/JUnit 5. Для изоляции зависимостей используются Mockito и Spring Security Test. Проверяются:

- хеширование пароля (BCrypt) и выдача JWT при регистрации;
- отказ при неверном пароле;
- правила доступа пользователя (только к своим фото для удаления);
- работа контроллеров фото, лайков, профилей и поиска;
- корректность HTTP-кодов ответа в позитивных и негативных сценариях;
- защита от двойного лайка через уникальный составной ключ (photo_id, user_id) в таблице photo_likes;
- генерация и проверка JWT-токенов.

Запуск тестов сервера:

```bash
./gradlew :server:test
```

Полная проверка серверного модуля вместе с JaCoCo verification:

```bash
./gradlew :server:check
```

## 4. JaCoCo

Для измерения покрытия Java-кода автоматическими тестами в серверном модуле применяется JaCoCo.

### 4.1. Подключение JaCoCo

В `server/build.gradle.kts` подключен Gradle-плагин JaCoCo:

```kotlin
plugins {
    id("jacoco")
}
```

Также указана версия инструмента:

```kotlin
jacoco {
    toolVersion = "0.8.12"
}
```

### 4.2. Генерация отчетов

После выполнения задачи `test` автоматически запускается `jacocoTestReport`. Поддерживаются два формата вывода:

- **HTML** — удобен для просмотра покрытия по пакетам, классам и строкам кода;
- **XML** — пригоден для CI/CD, внешних анализаторов и публикации метрик.

Команда запуска:

```bash
./gradlew :server:test :server:jacocoTestReport
```

Основные файлы отчетов:

```text
server/build/reports/jacoco/test/html/index.html
server/build/reports/jacoco/test/jacocoTestReport.xml
```

### 4.3. Порог покрытия

В проекте установлен минимальный порог покрытия серверного модуля — 40%. Контроль осуществляется задачей `jacocoTestCoverageVerification`, от которой зависит задача `check`. При падении покрытия ниже порога сборка серверного модуля завершается с ошибкой.

Команда проверки покрытия:

```bash
./gradlew :server:jacocoTestCoverageVerification
```

### 4.4. Как использовался отчет JaCoCo

Отчёт использовался для решения следующих задач:

- выявление классов, ещё не покрытых тестами;
- мониторинг критически важных участков серверной логики: аутентификация, фото, лайки, безопасность;
- контроль того, что новые изменения не снижают покрытие ниже установленного порога;
- планирование дальнейших тестов: в первую очередь покрывались классы с бизнес-логикой и ветвлениями, затем контроллеры и вспомогательные компоненты.

## 5. Postman

Postman применялся для ручного и интеграционного тестирования REST API серверной части. В отличие от unit-тестов, Postman позволяет проверить API в условиях, максимально приближенных к реальному использованию клиентским приложением: с фактическим URL, HTTP-методом, JSON-телом, заголовком `Authorization` и реальным ответом сервера.

### 5.1. Подготовка окружения

Для локального запуска API использовался сервер на порту `8080`. Базовый адрес в Postman:

```text
http://localhost:8080
```

Рекомендуемые переменные окружения Postman:

| Переменная | Назначение | Пример |
| --- | --- | --- |
| `baseUrl` | Базовый URL сервера | `http://localhost:8080` |
| `token` | JWT пользователя или администратора | `eyJ...` |
| `userId` | ID текущего аккаунта | `1` |
| `photoId` | ID избранного города | `10` |

Для защищенных запросов добавлялся заголовок:

```http
Authorization: Bearer {{token}}
```

### 5.2. Коллекция запросов Postman

Коллекция запросов была организована по функциональным группам API:

#### Auth

- `POST {{baseUrl}}/api/auth/register` — регистрация пользователя;
- `POST {{baseUrl}}/api/auth/login` — вход и получение JWT.

Пример тела регистрации:

```json
{
  "email": "user@example.com",
  "fullName": "Иван Иванов",
  "password": "secret123"
}
```

После успешного ответа JWT сохранялся в переменную `token`, а идентификатор пользователя — в `userId`.

### Photos
- `GET {{baseUrl}}/api/photos` — получить все фотографии;
- `GET {{baseUrl}}/api/photos/{{photoId}}` — получить фотографию по ID;
- `GET {{baseUrl}}/api/photos/search?query=...` — поиск фотографий;
- `GET {{baseUrl}}/api/photos/files/{{filename}}` — получить загруженное фото;
- `GET {{baseUrl}}/api/photos/by-author?authorId=...` — получить фото по автору;
- `POST {{baseUrl}}/api/photos` — создать новую фотографию (JSON);
- `POST {{baseUrl}}/api/photos/upload` — загрузить фото и получить URL (Multipart);
- `POST {{baseUrl}}/api/photos/{{photoId}}/like` — поставить/убрать лайк;
- `PUT {{baseUrl}}/api/photos/{{photoId}}` — обновить фотографию;
- `DELETE {{baseUrl}}/api/photos/{{photoId}}` — удалить фотографию.

Пример тела запроса создания фото:

```json
{
  "title": "Закат в горах",
  "description": "Красивый пейзаж",
  "imageUrl": "https://example.com/sunset.jpg"
}
```

#### Users

- `GET {{baseUrl}}/api/users` — получить список всех пользователей;
- `GET {{baseUrl}}/api/users/{{userId}}` — получить публичный профиль пользователя по ID;
- `GET {{baseUrl}}/api/users/search?query=...` — поиск пользователей по имени или email;
- `GET {{baseUrl}}/api/users/me` — получить данные текущего пользователя;
- `PUT {{baseUrl}}/api/users/profile` — обновить данные профиля.

### 5.3. Виды проверок через Postman и Swagger

Через Postman и Swagger UI выполнялись следующие категории проверок:

- **позитивные сценарии**: успешная регистрация, вход, создание фото, загрузка файла, постановка лайка, получение списков;
- **негативные сценарии**: некорректный пароль, запрос без токена, запрос с токеном другого пользователя, попытка удалить чужое фото;
- **проверка HTTP-кодов**: `200 OK`, `201 Created`, `204 No Content`, `400 Bad Request`, `401 Unauthorized`, `403 Forbidden`, `404 Not Found`;
- **проверка структуры JSON**: наличие полей `id`, `title`, `description`, `imageUrl`, `likesCount`, `likedByCurrentUser`, `authorFullName`;
- **проверка изменения состояния**: после `POST` объект появляется в списке, после `DELETE` объект отсутствует, счётчик лайков изменяется;
- **проверка загрузки файлов**: корректная обработка Multipart-запросов, сохранение файлов на сервере.

### 5.4. Примеры Postman-скриптов

Для запросов аутентификации применяется скрипт, автоматически сохраняющий токен:

```javascript
pm.test("response is successful", function () {
    pm.response.to.have.status(200);
});

const json = pm.response.json();
pm.environment.set("token", json.token);
pm.environment.set("userId", json.userId);
```

Для защищённых запросов проверяется принадлежность данных текущему пользователю:

```javascript
pm.test("photo belongs to current user", function () {
    const json = pm.response.json();
    pm.expect(json.authorId).to.eql(Number(pm.environment.get("userId")));
});
```

## 6. Итоговый процесс проверки

Перед завершением каждого этапа разработки применялся следующий порядок действий:

1. Обновить или дополнить сценарии в тест-плане.
2. Реализовать или актуализировать автотесты для воспроизводимых проверок.
3. Запустить тесты мобильного приложения при изменениях в клиентской логике.
4. Запустить тесты сервера и проверку JaCoCo при изменениях в backend.
5. Выполнить ключевые API-сценарии в Postman/Swagger UI, если изменялись контроллеры, DTO, безопасность или формат ответов.
6. Зафиксировать результат: какие проверки пройдены, какие дефекты обнаружены, что требует повторной проверки.

Такой подход позволил объединить планирование тестирования, автоматическую защиту от регрессий и ручную валидацию реального REST API в единую систему контроля качества проекта.

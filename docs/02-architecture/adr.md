# Архитектурные решения (ADR)

В этом документе зафиксированы ключевые архитектурные решения, принятые в ходе проектирования.

## ADR-001: Выбор архитектуры PCMEF и MVVM

- **Статус:** Принято
- **Контекст:** Необходимо обеспечить четкое разделение ответственности в мобильном приложении, облегчить тестирование и поддержку кода. Требуется соответствие требованиям курсового проекта (ПКМЕФ).
- **Решение:** Использовать паттерн PCMEF, адаптированный под MVVM (Model-View-ViewModel) от Google. 
  - Presentation = Compose UI
  - Control = ViewModel
  - Mediator = Repository (реализован в `/foundation/repository/`)
  - Entity = Domain Models
  - Foundation = Room/Retrofit
- **Последствия:** Упрощается навигация и тестирование ViewModel. Состояние UI инкапсулировано и переживает конфигурационные изменения (поворот экрана).

## ADR-002: Локальное кэширование через Room (Single Source of Truth)

- **Статус:** Принято (упрощенная реализация)
- **Контекст:** Приложение должно работать в оффлайн-режиме (Требование 3.3.5 методички). Прямые запросы к сети из UI приводят к ошибкам при отсутствии интернета и дублированию кода.
- **Решение:** Реализовать стратегию "Единый источник истины" (Single Source of Truth). UI (ViewModel) всегда слушает только локальную базу данных Room через `Flow`. Репозиторий (Mediator) пытается обновить БД из сети в момент обновления пользователем.
- **Реализованные аспекты:**
  -  UI слушает только локальную базу данных через Flow
  -  Стратегия `OnConflictStrategy.REPLACE` для автоматического кэширования
  -  Graceful fallback на кэш при ошибке сети
- **Последствия:** Полная поддержка чтения из кэша. UI всегда отображает актуальные локальные данные. Ограничение: при отсутствии сети фото загружаются оптимистично, но при ошибке отменяются.

## ADR-003: Асинхронность через Kotlin Coroutines и Flow

- **Статус:** Принято
- **Контекст:** Сетевые запросы и операции с БД являются блокирующими. Использование колбэков приводит к "Callback Hell" и усложняет обработку ошибок.
- **Решение:** Использовать Kotlin Coroutines для `suspend` функций (Retrofit, Room) и Kotlin `Flow` для реактивной передачи данных из Repository в ViewModel.
- **Последствия:** Чистый, линейный асинхронный код. Легкая отмена запросов при уничтожении ViewModel (`viewModelScope`).

## ADR-004: Внедрение зависимостей (DI) вручную

- **Статус:** Принято
- **Контекст:** Необходимо передавать экземпляры `PhotoApi` и `PhotoDao` в `PortfolioRepository`, а `Repository` в `ViewModel`. Использование Hilt/Dagger избыточно для текущего объема проекта и усложняет порог входа.
- **Решение:** Использовать ручной DI через `ViewModelFactory` и передачу `Context` из `MainActivity`.
- **Последствия:** Полный контроль над созданием объектов. Отсутствие магии и скрытых зависимостей. Легкость отладки.
